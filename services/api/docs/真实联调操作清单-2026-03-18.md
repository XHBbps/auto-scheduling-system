# 真实联调操作清单（2026-03-18，已按正式用户认证更新）

## 一、先确认当前认证方式

当前正式链路：
- 登录：`POST /api/auth/login`
- 会话查询：`GET /api/auth/session`
- 退出：`POST /api/auth/logout`
- 认证介质：浏览器或客户端保存的 `user_session` Cookie

联调时统一按“登录 -> 带 Cookie 访问受保护接口 -> 退出”的链路检查，不需要额外自定义认证请求头。

## 二、最小联调步骤

1. 打开登录页并使用账号密码登录。
2. 观察浏览器是否收到会话 Cookie。
3. 登录后进入以下受保护页面验证：
   - 销售计划表
   - BOM 数据
   - 排产日历
   - 数据同步
   - 用户管理
4. 刷新受保护页面，确认仍保持已登录状态。
5. 退出登录后再次访问受保护页面，确认会被正确拦回登录页。

## 三、接口联调检查点

### 1. 登录
- 请求：`POST /api/auth/login`
- 入参：`username`、`password`
- 预期：返回当前用户、角色、过期时间，并下发 Cookie

### 2. 会话探活
- 请求：`GET /api/auth/session`
- 预期：已登录时返回 `authenticated = true`

### 3. 受保护接口
- 请求示例：`GET /api/admin/users`
- 预期：未登录返回 401；无角色返回 403；管理员可正常访问

## 四、排查顺序

1. 先看登录接口是否成功。
2. 再看 Cookie 是否写入浏览器。
3. 再看刷新后 `GET /api/auth/session` 是否仍返回已登录。
4. 最后再看具体页面请求是否因为 401 / 403 被路由守卫拦回。
