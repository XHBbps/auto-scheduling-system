# 自动排产系统 P0-P2 历史清单（2026-03-19）

> 记录日期：2026-03-19
> 说明：本文档保留 2026-03-19 阶段的历史治理清单，已按正常中文重写，便于后续追溯当时的重点问题与处理顺序。

## 1. 当时阶段结论

当时项目已具备可联调的 MVP 骨架，但距离稳定上线还有明显差距。
核心判断是：
- 主链路已成型
- 真实生产数据联调已开始
- 安全、状态、部署和工程治理仍需集中收口

## 2. 当时识别的 P0 项

| 编号 | 问题 | 风险 | 当时建议 |
|---|---|---|---|
| P0-1 | 管理接口依赖前端静态 Token | 安全面不足，存在泄露风险 | 先做最小正式鉴权方案，至少完成服务端统一鉴权收口 |
| P0-2 | BOM 缺失只停留在报错层 | 订单可能卡在排产前，无法形成治理闭环 | 建立缺失 BOM 队列、补数和状态回写机制 |
| P0-3 | 动态排产状态未形成稳定视图 | 总览、列表与实际排产就绪度可能不一致 | 引入 snapshot 作为状态真源，并支持刷新收口 |
| P0-4 | Docker / 数据库 / 启动链路未完全收口 | 环境切换与联调成本高 | 用 Compose 固化依赖，并确保迁移与启动顺序可控 |
| P0-5 | 批量排产事务边界不清晰 | 容易产生半成品脏数据 | 强化按订单隔离与失败回滚 |
| P0-6 | 数据库迁移基线不稳定 | 环境一致性差 | 收口 Alembic baseline 与迁移流程 |

## 3. 当时识别的 P1 项

| 编号 | 问题 | 风险 | 当时建议 |
|---|---|---|---|
| P1-1 | 查询与导出存在重复查询风险 | 数据量上来后性能下滑 | 优先优化高频列表、导出与详情链路 |
| P1-2 | Postgres 与 SQLite 行为差异 | 本地验证与真实环境结论可能不一致 | 尽快转向 Postgres 作为主验证环境 |
| P1-3 | 外部同步缺少统一超时 / 重试策略 | 外部接口抖动时稳定性差 | 统一 timeout、retry、日志与失败记录 |
| P1-4 | Compose 与环境变量仍偏临时态 | 部署收口不足 | 明确各服务依赖、环境变量和健康检查 |

## 4. 当时识别的 P2 项

| 编号 | 问题 | 风险 | 当时建议 |
|---|---|---|---|
| P2-1 | 文档与事实不同步 | 新窗口接手成本高 | 同步项目进度文档与阶段摘要 |
| P2-2 | 缺少 CI / 自动验证 | 回归风险高 | 增加最小构建、测试与迁移校验 |
| P2-3 | 工程卫生不足 | 容易引入历史噪音 | 清理乱码、临时文件和目录命名问题 |

## 5. 历史执行顺序建议

### 第一阶段
1. 收口动态状态真源与 snapshot 刷新机制
2. 完成 BOM 缺失治理闭环
3. 修复一键排产、导出错位、删除接口等明确运行时问题

### 第二阶段
1. 收口 Docker、数据库与迁移流程
2. 收口同步任务、巡检、失败重试和日志可观测性
3. 优化高频查询、导出与详情链路

### 第三阶段
1. 清理文案乱码与历史命名残留
2. 补齐前后端最小测试能力
3. 继续推进权限、安全和工程化治理
