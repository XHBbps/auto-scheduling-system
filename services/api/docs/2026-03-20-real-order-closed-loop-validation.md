# 自动排产系统真实订单闭环验证（2026-03-20 历史版）

> 记录日期：2026-03-20
> 说明：本文档保留 2026-03-20 阶段真实订单联调验证的历史结论，内容已按正常中文重写。

## 1. 验证目标

当时验证的核心目标有三项：
1. 使用真实数据库与真实订单样本验证整机排产链路是否可跑通
2. 确认动态状态判断与问题单是否能反映真实订单情况
3. 识别从真实联调走向稳定交付前仍需补齐的关键能力

## 2. 当时的验证环境

- 数据库：PostgreSQL
- 运行方式：Docker / 本地联调混合方式
- 验证对象：真实订单样本、真实 BOM 情况、真实同步后数据
- 关键链路：
  - `POST /api/admin/schedule/run`
  - snapshot 刷新
  - BOM 缺失识别
  - 问题单生成

## 3. 当时得到的主要结论

### 3.1 主链路可以跑通
- 系统已经具备从销售计划、BOM、基准到排产结果的主链路能力
- 对部分真实订单可以完成整机排产并写入结果
- 说明项目已经不只是 demo，而是具备真实业务试跑基础

### 3.2 真实联调暴露出状态与数据口径问题
- 部分订单会因为缺少 BOM、待发图、待触发等原因无法进入稳定排产
- 当时这些状态还没有完全收口到统一真源
- 导致页面展示、手工排产、异常治理之间存在口径不一致风险

### 3.3 BOM 缺失是当时的重点卡点之一
- 部分真实订单缺少完整 BOM
- 这会直接影响排产前置校验、零件排产和总览口径
- 因此当时明确提出要把 BOM 缺失从“报错”升级为“治理闭环”

### 3.4 环境与工程问题仍明显存在
- Docker 与数据库依赖需要进一步规范
- 数据库迁移基线与环境一致性需要加强
- 历史命名、乱码与文档同步问题会增加后续协作成本

## 4. 当时沉淀出的行动项

### 高优先级
1. 用 snapshot 承接动态状态真源
2. 建立 BOM 缺失队列、补数与重试机制
3. 修复一键排产、导出、删除接口等高优先级缺陷

### 中优先级
1. 统一 Docker / 数据库 / Alembic 流程
2. 提高高频接口查询与导出性能
3. 补齐同步巡检、失败重试和可观测性

### 持续项
1. 清理乱码与历史脏文案
2. 同步进度文档与项目结论
3. 为后续权限、安全和测试治理做准备
