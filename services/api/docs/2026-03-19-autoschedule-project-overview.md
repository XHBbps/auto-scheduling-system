# 自动排产系统项目总览（2026-03-19 历史版）

> 记录日期：2026-03-19
> 说明：本文档用于保留项目在 2026-03-19 阶段的总体判断与架构认知，内容已按正常中文重写。

## 1. 当时总体判断

当时的项目结论是：

**自动排产系统已经从 demo 进入“可联调、可治理”的阶段，但还没有完成稳定生产化收口。**

主要原因：
- 前后端骨架已完整
- 外部数据同步、基准维护、排产、查询、导出主链路已具备
- 但安全、动态状态、部署、数据库一致性和性能治理仍需继续推进

## 2. 当时的系统结构

### 2.1 前端
- Vue 3 + TypeScript + Vite + Element Plus
- 包含排产总览、排产列表、零件排产、异常管理、同步控制台、同步日志、基准配置、外源数据等页面
- 具备基本查询、导出、排序与展示能力

### 2.2 后端
- FastAPI + SQLAlchemy Async + PostgreSQL / Alembic
- 分层结构：router / service / repository / scheduler / sync
- 主要模块：
  - 销售计划同步
  - BOM 同步
  - 生产订单同步
  - 研究所 / 发图状态同步
  - 基准维护
  - 整机 / 零件排产
  - 总览、列表、详情、导出、异常、巡检

### 2.3 数据主线
1. 外部数据进入原始表
2. 排产与问题单形成业务结果
3. snapshot 负责聚合排产状态与就绪度信息
4. 前端优先围绕 snapshot、排产结果和问题单进行联查与展示

## 3. 当时识别出的核心问题

### 3.1 状态模型问题
- 没有稳定承接“待发图 / 待触发 / 可排产 / 缺少 BOM”的统一真源
- 已排产结果、动态校验结果和问题单之间存在口径分散
- 因此总览、列表和手动排产入口容易出现不一致

### 3.2 数据治理问题
- BOM 缺失只报错，不形成闭环治理
- 同步失败、部分数据不一致时缺少稳定补救路径
- 外部接口抖动时缺少统一 timeout / retry / observability

### 3.3 工程化问题
- 前端权限仍是临时方案
- Docker / Compose / 数据库迁移流程未完全固化
- 历史命名、乱码、文档和目录结构仍比较粗糙

## 4. 当时建议的推进方向

### 第一优先级
1. 建立 snapshot 真源与动态状态刷新机制
2. 建立 BOM 缺失治理闭环
3. 修复高优先级运行时缺陷

### 第二优先级
1. 收口 Docker / 数据库 / Alembic
2. 收口同步任务、巡检、重试与可观测性
3. 优化高频查询与导出性能

### 第三优先级
1. 清理乱码与历史命名
2. 补齐自动化测试能力
3. 逐步推进权限、安全和工程规范
