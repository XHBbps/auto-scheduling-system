## P0 数据模型与并发治理
- [x] 为 `part_cycle_baseline` 增加工厂 `plant`、样本数 `sample_count`、源数据更新时间 `source_updated_at` 字段。
- [x] 为自动入库主键范围补齐唯一约束与索引，确保“零件类型 + 机型 + 工厂”不会重复写入，并补齐空工厂兜底唯一性。
- [x] 将零件周期基准写入链路收口为仓储层统一处理，正式消除“先查再插”带来的并发重复风险。
- [x] 手工维护值与历史回算值分流治理：自动回算不得覆盖 `cycle_source = manual` 的正式记录。

## P1 自动重建正式入库
- [x] 重写零件周期基准重建服务，按“零件类型 + 机型 + 工厂”分组统计中位周期、单件周期、样本数与波动率。
- [x] 自动正式入库仅保留满足阈值的历史组：`sample_count >= 3` 且 `max_cycle / min_cycle <= 2`。
- [x] 不新增候选结果表；未达标分组只做统计与日志输出，不生成冗余业务数据。
- [x] 自动重建对历史来源采用数据库级 upsert；未变化记录不重复写入。
- [x] 自动重建结束后清理不再达标或已失效的历史来源正式值，但不影响人工维护记录。
- [x] 新增后台任务类型 `part_cycle_baseline_rebuild`，使用固定 `dedupe_key`，确保同一时刻只有一个零件周期基准重建任务运行。
- [x] 生产订单同步成功后自动入队零件周期基准重建任务，形成“同步完成 -> 异步重建”的闭环。
- [x] 新增手动触发接口 `POST /api/admin/part-cycle-baselines/rebuild`，只负责入队，不在接口请求内同步重算。
- [x] 自动重建仅在正式值发生新增、更新、失效时按影响范围批量刷新 snapshot，禁止逐条刷新。

- [x] 前端“零件周期基准”页面补齐工厂、样本数展示与筛选能力。
